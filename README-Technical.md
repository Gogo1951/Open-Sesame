# Open Sesame — Technical Reference

This document combines architecture notes and contribution guidance for developers working on Open Sesame. For end-user documentation, see [README.md](https://github.com/Gogo1951/Open-Sesame/blob/main/README.md).

The addon namespace is `ns` — every file begins `local ADDON_NAME, ns = ...`, and everything modules share hangs off that table. The only globals the addon itself defines are the `OpenSesameDB` SavedVariable, the named `OS_ScanTooltip` scanning frame (plus the `OS_ScanTooltipTextLeft*` font strings it spawns), and the `SLASH_OPENSESAME1` / `SlashCmdList.OPENSESAME` pair the WoW slash API requires for `/os`; the minimap button is a global frame owned by LibDBIcon.

Load order is set by the `.toc`: libraries first, then `Locales/`, then `Data/` (which builds `ns.HEX`, the item tables, and constants), then `Features/`, then `Options/`. Files that build state at load time (`ns.COLORS`, `ns.BRAND_PREFIX`) depend on this order — `Data.lua` and `Utilities.lua` must run before `Announcements.lua`.

## Architecture

### Event Loop

Core.lua owns the single event frame. Handlers live in the `EventHandlers` table, keyed by event name; the frame registers every key at load (guarded by `C_EventUtils.IsEventValid` where available, so one bad event name can't abort the whole loop on a future build). `ns.EVENT_NAMES` is exported from that table — sorted — so the Diagnostics event probe can never drift from what the addon actually registers. The dispatcher also forwards every event to `ns:LogEvent` while the diagnostics event log is recording.

Events the addon listens for:

- **`PLAYER_LOGIN`** — initialize `OpenSesameDB`, merge defaults, mirror `autoOpen`/`speedyLoot` into the runtime flags, derive `minimap.hide` from `showMinimap`, init the minimap button, print the welcome message.
- **`PLAYER_ENTERING_WORLD`** — on initial login or `/reload`, wait `ns.WORLD_LOAD_DELAY` (8s) for bag/loot APIs to settle, then run a forced scan and ensure Auto Loot is on (whenever Auto-Opening *or* Speedy Loot is enabled). On zone changes, just enter a 2s "quiet" window to suppress chat noise.
- **`BAG_UPDATE_DELAYED`** / **`BAG_NEW_ITEMS_UPDATED`** — debounced `ScheduleScan()` (the soft form). Fires often; debouncing collapses bursts.
- **`LOOT_READY`** / **`LOOT_OPENED`** — inspect the open loot slots' source GUIDs (`GetLootSourceInfo`) and, when the loot comes from a world corpse/chest/object (`Creature-`/`Vehicle-`/`GameObject-`) rather than an item, stamp `lastWorldLootAt` — the timestamp that gates the loot sound in `CHAT_MSG_LOOT`. Item-sourced loot (disenchant, prospect/mill, container opens, item merges) carries an `Item-` source and instead *clears* the stamp, so it can't ride a real corpse's sound window even if one was set moments earlier. When the source is still unknown — `GetLootSourceInfo` missing, the window empty, or the GUIDs not yet populated — the stamp is left untouched, so a late-arriving world GUID on `LOOT_OPENED` still counts. `LOOT_READY` then also calls `ns.HandleSpeedyLoot()` (Speedy-Loot.lua); the stamp runs first so the source GUIDs are read before Speedy Loot empties the slots — see Speedy Loot.
- **`CHAT_MSG_LOOT`** — match the localized self-loot prefixes (derived from `LOOT_ITEM_SELF` / `LOOT_ITEM_PUSHED_SELF`). If the looted item is in `AllowedItems` with value `false` (a lockbox), announce it'll auto-open on unlock — gated on both `autoOpen` and `lockboxNotifications`, see Lockbox Notifications; optionally play the rare-loot sound, but only for Uncommon+ items looted from a world corpse/chest within `ns.LOOT_SOUND_WINDOW` (1s) of the `lastWorldLootAt` stamp. Always requests a debounced rescan.
- **`PLAYER_REGEN_ENABLED`** / **`UPDATE_STEALTH`** — re-scan once combat ends or stealth drops.
- **`UNIT_SPELLCAST_SUCCEEDED`** — when the player casts Pick Lock (spell ID 1804), schedule a forced scan after `ns.PICK_LOCK_RESCAN_DELAY` (0.5s) so the newly-unlocked container queues.
- **`UI_ERROR_MESSAGE`** — detect "inventory full" (plain-text match against `ERR_INV_FULL`, or numeric error code), pause the addon, play the race/gender bag-full sound, and rate-limit re-triggers via `ns.BAG_FULL_COOLDOWN` (10s).
- **`BANKFRAME_CLOSED`, `GOSSIP_CLOSED`, `MAIL_CLOSED`, `MERCHANT_CLOSED`, `TRADE_CLOSED`, `QUEST_FINISHED`** + **`PLAYER_INTERACTION_MANAGER_FRAME_HIDE`** — forced re-scan once the player exits a UI interaction.
- **`LOADING_SCREEN_ENABLED` / `_DISABLED`** — extend the quiet window across loading screens.

`ns.EnsureAutoLoot()` also runs when the player switches Auto-Opening or Speedy Loot **on** (minimap button or options), so either feature guarantees Auto Loot without a reload.

### Combat & Interaction Gating

`IsSafeToOpen()` is the single guard the open pipeline checks before every action. It bails on:

- `not ns.isEnabled` or `ns.isPaused`
- `UnitAffectingCombat("player")`
- `IsInteractionActive()` — any of `MerchantFrame`, `MailFrame`, `TradeFrame`, `BankFrame`, `GuildBankFrame`, `AuctionFrame`, `GossipFrame`, `QuestFrame`, or `StaticPopup1` shown
- Stealth or Shadowmeld (spell ID 20580; checked separately because `IsStealthed` doesn't return true under Shadowmeld)
- An active cast/channel via `UnitCastingInfo` / `UnitChannelInfo`
- A live `GameTooltip` showing an item — opening would dismiss the tooltip mid-comparison
- Free bag slots below `ns.MIN_FREE_SLOTS` (4)

There is no separate combat queue — the bag queue *is* the deferral mechanism. When `IsSafeToOpen()` is false, `OpenTick()` stops rescheduling itself and deferred items simply sit in their bag slots. Work replays through the events that signal "safe again": `PLAYER_REGEN_ENABLED` (combat ended) and `UPDATE_STEALTH` (stealth dropped) both force a fresh scan that rebuilds the queue and restarts the tick. The one exception is an active cast/channel: `OpenTick()` reschedules *itself* and waits rather than dropping, because a cast is short-lived and has no "ended" event the pipeline already listens for.

### Scan → Queue → Open pipeline

Owned entirely by Core.lua. Three phases:

1. **Scan** — `ScheduleScan(force?)` runs immediately when forced, otherwise debounces via `C_Timer.After(ns.SCAN_DEBOUNCE)` (0.5s); bursts collapse to one scan. Pause/resume flips here through `ShouldPause()` (asymmetric thresholds: pause below `MIN_FREE_SLOTS`, resume only at `MIN_FREE_SLOTS + 1` — hysteresis that avoids flapping when a single slot opens and closes). When Auto-Opening is disabled the scan returns before building the queue, so a disabled addon does no bag work.

2. **Queue** — `BuildQueue()` walks bags 0–4, reads each slot's item ID via `SafeFastItemID()` (which parses `item:%d+` out of the link when `GetContainerItemID` returns nil), and pushes `(bag, slot, itemId)` triples into a flat array-backed queue. `AllowedItems[itemId] == true` queues unconditionally; `== false` queues only when `IsItemLocked(bag, slot)` is false (the lockbox has been picked open).

3. **Open** — `OpenTick()` fires every `ns.OPEN_TICK_INTERVAL` (0.25s). It pops one item, re-checks the slot still holds the same item ID (so it never opens whatever the player just dragged in), calls `ns.UseContainerItem(bag, slot)`, then schedules a re-check `ns.OPEN_RECHECK_DELAY` (0.25s) later: if the item is still present and still allowed, it requeues it (some containers take several uses to empty). The tick reschedules itself only while the queue has work and `IsSafeToOpen()` holds.

The `ns.state.openTimerLive` flag exists because `C_Timer.After` callbacks can't be cancelled — it prevents double-scheduling the tick when a scan completes mid-flight.

`IsItemLocked()` uses a hidden scanning tooltip (the named `OS_ScanTooltip` frame) anchored to `WorldFrame`, scanning each line for the localized `LOCKED` global. Anchoring to `WorldFrame` keeps it clear of `GameTooltip:GetOwner()` checks elsewhere.

### Status Output

All player-facing text routes through Announcements.lua. `ns.PrintMessage` prepends `ns.BRAND_PREFIX` (built once at load from `ns.COLORS` + `L["ADDON_TITLE"]`) and writes to `DEFAULT_CHAT_FRAME`. `ns.StatusPrint` is a deduped channel layered on top of it for transient state changes:

- **Quiet window** — `ns.SetQuiet(seconds)` pushes `ns.state.quietUntil` forward; `StatusPrint` drops anything fired while quiet. Login, `/reload`, zoning, and loading screens all open a quiet window so the player isn't spammed during churn.
- **5-second dedup** — `StatusPrint` suppresses a message identical to the last one within 5s (`ns.state.lastStatusMsg` / `lastStatusAt`), so repeated pause/resume flips collapse to one line.
- **Held during interactions** — pause/resume prints are withheld while a merchant/mail/bank/etc. window is open (`AnnounceStatus` early-returns on `IsInteractionActive`), otherwise "Resumed" would be lost in the player's vendoring. `ns.state.announcedPaused` tracks what the player was last told; `OnInteractionClosed` flushes a held message after `ns.STATUS_FLUSH_DELAY` (0.25s) once the window closes.

### Speedy Loot

Speedy-Loot.lua exposes `ns.HandleSpeedyLoot()`, called from Core's `EventHandlers:LOOT_READY` so a single registration owns the event — Core stamps world-loot state first, then runs this. On call:

1. Bail if `ns.isSpeedyLoot` is false.
2. Compute `shouldAutoLoot = autoLoot ~= IsModifiedClick("AUTOLOOTTOGGLE")` — respects the auto-loot CVar and any in-progress modifier override (Shift to invert); bail if false.
3. Rate-limit via `ns.LOOT_DELAY` (0.25s) against rapid duplicate fires.
4. Bail if `GetNumLootItems()` is zero.
5. **Stand down entirely as master looter** (`lootMethod == "master" and masterLooterPartyID == 0`) — see Common Pitfalls.
6. Read general-bag free slots via `ns.GetFreeSlots()`; if bags are full and real item slots are waiting, leave the window visible and loot nothing.
7. Hide `LootFrame`, then iterate slots in reverse calling `LootSlot()`: money/currency always loot (no bag cost); item slots loot while free slots remain, decrementing as they go.

Two guards inside the loop:

- **Ignore list**: items in `ns.IgnoreItems` (e.g., quality-graded gem sacks, BoP grab bags) are skipped and the loot frame is re-shown. The "open manually" notice is told once per item per 5s via `ns.state.recentAnnouncements`, and only when both `autoOpen` and `lockboxNotifications` are on (see Lockbox Notifications) — the skip itself always happens regardless of the notification setting.
- **Bags filling mid-loop**: once free slots hit zero, remaining item slots are left and the loot frame is re-shown rather than stranding loot behind a hidden window.

Auto-loot and loot-method detection both use availability-based picks (`if C_CVar.GetCVarBool then … else …`, `if GetLootMethod then … elseif C_PartyInfo.GetLootMethod then …`) rather than `(modern() or legacy())`, which would fall through to the legacy path whenever the modern call legitimately returns `false`/an enum. Keep the `if`-form. `C_PartyInfo.GetLootMethod` returns the method as an enum number; `GetLootMethodCompat` maps enum `2` (or `Enum.LootMethod.MasterLoot` where present) back to the legacy `"master"` string so the master-looter guard reads consistently across clients.

### Minimap Button

Minimap-Button.lua registers a single LibDataBroker launcher (`ns.InitMinimap`, called from `PLAYER_LOGIN`) and hands its `minimap` subtable to `LDBIcon:Register`. The click handlers map to the three feature toggles — left-click Auto-Opening, right-click Speedy Loot, middle-click Loot Sounds — and the tooltip reflects each feature's current state.

`ns.UpdateMinimapIcon` swaps the icon and broker text between three states derived from the runtime flags: `off` (not enabled), `paused` (enabled but out of bag space), and `on`. Core calls it after every state change.

Visibility is a separate concern from icon state. `showMinimap` (SavedVariables) is the source of truth; `ns.SetMinimapShown(shown)` writes it and mirrors the inverse onto `minimap.hide` — the field LibDBIcon actually reads — then calls `LDBIcon:Show` / `LDBIcon:Hide`. `PLAYER_LOGIN` re-derives `hide` from `showMinimap` at load, and a settings reset restores `showMinimap` to on. The `minimap` subtable that stores the button's position is never wiped, so neither resets nor visibility toggles ever move the button.

### Lockbox Notifications

Two chat notices explain how a locked container will be handled: the **auto-open-on-unlock** notice (`CHAT_MSG_LOOT` in Core.lua, fired when a looted item is in `AllowedItems` with value `false`) and the **open-manually** notice (Speedy-Loot.lua, fired when an item on the ignore list is skipped mid-loot). Both are gated on **two** flags — `ns.DB.autoOpen` *and* `ns.DB.lockboxNotifications` — so a player who has turned Auto-Opening off hears nothing about lockboxes even with notifications still enabled, and so the notices stay silent when the player opts out specifically.

The options UI mirrors the same dependency: the **Enable Lockbox Notifications** toggle is `hidden` whenever `autoOpen` is false, so the checkbox only appears while Auto-Opening is on. Keep the `autoOpen and lockboxNotifications` pair on *both* call sites; dropping it from one makes the runtime and the UI disagree.

### Options Panel & Slash Command

Options.lua registers the two AceConfig panels — General (`ns.BuildGeneralOptions`) and Diagnostic Tools (`ns.BuildDiagnosticsOptions`, registered last so it nests at the bottom of the tree) — and binds `/os` to open the General panel. `OpenOptions` resolves the panel three ways for cross-client coverage: `Settings.OpenToCategory` (the retail-derived Settings API) first, then the legacy `InterfaceOptionsFrame_OpenToCategory` (called twice — a long-standing Blizzard quirk where the first call only selects the parent), then `AceConfigDialog:Open` as a last resort.

The General panel (Options-General.lua) is built from the shared widget helpers in Options-Utilities.lua, and every section follows the same shape: spacer → header → spacer → description → spacer → toggle. Section ordering is driven by the integer `order` fields, not table position — the `/Commands` block sits first (orders 10–13), then Auto-Opening (20s), Speedy Loot (30s), Loot Sounds (40s), Reset (80s), Feedback (90s). Toggle labels read from dedicated `OPTIONS_ENABLE_*` strings so the section headers (`AUTO_OPENING`, etc.) can stay plain while the checkboxes say "Enable …".

### Item Allowlist

`ns.AllowedItems` in Data.lua is a flat `{ [itemId] = boolean }` map; the boolean carries meaning, not just presence:

- **`true`** — open immediately. Clams, gift bags, fishing junk, quest containers — anything that doesn't need a key or lockpick.
- **`false`** — requires unlock. Lockboxes and similar; the pipeline queues these only once `IsItemLocked()` returns false (a rogue or key has popped them).

`ns.IgnoreItems` is a presence-only `{ [itemId] = true }` map used by Speedy Loot to skip auto-looting items that would otherwise prompt a Bind-on-Pickup confirmation mid-loot.

Both tables are grouped by expansion with comment dividers (`-- 01. World of Warcraft`, `-- 02. … The Burning Crusade`, `-- 03. … Wrath of the Lich King`) and alphabetized by item name within each block. They were curated by hand, not generated from a SQL dump, so there's no source query to regenerate from.

### Diagnostic Tools

Off by default behind `ns.diagnostics.enabled`, a runtime-only flag (never a SavedVariable) that resets to off on every reload — file-scope init is correct here because the "initialize on PLAYER_LOGIN" rule applies only to SavedVariables. `ns:SetDiagnosticsEnabled(false)` also stops the event log. Features/Diagnostics.lua builds plain-text reports on demand — never on load or panel open — and Options/Options-Diagnostics.lua renders the panel, registered last so it sits at the bottom of the settings tree. When the gate is off, every widget below the enable toggle is `hidden`.

Diagnostics strings live in `ns.DiagnosticsStrings` (Features/Diagnostics.lua) as plain English and are **intentionally not localized** — they are developer-facing troubleshooting text. The only localized value the panel reads is the addon's own display name (`ns.L["ADDON_TITLE"]`).

The panel exposes eight tools:

- **Event Log** — a live capture, detailed below.
- **Event Registration** — for each name in `ns.EVENT_NAMES`, reports `C_EventUtils.IsEventValid` plus a register/unregister round-trip on a throwaway probe frame (no handler attached, so nothing is processed).
- **API Endpoints** — `ns.DIAGNOSTIC_API_CHECKS`, existence/shape only; modern and legacy fallbacks are listed separately so the report shows exactly what each client provides. Kept aligned with the API guards in Utilities/Core/Speedy-Loot.
- **Loot Method** — prints the raw return values of `GetLootMethod` / `C_PartyInfo.GetLootMethod` and how the Speedy Loot master-looter guard interprets them, because an existence check can't prove the `(method, partyMasterID)` shape.
- **Other Add-ons** — name/version/loadable for every installed add-on.
- **Saved Variables** — a sorted recursive dump of `OpenSesameDB` (depth-capped at 8).
- **Library Versions** — `LibStub:IterateLibraries()` with minor versions.
- **Taint Log** — toggles the `taintLog` CVar (0/2). This is the **only** state any probe writes; the External Tools section below it just points at `/console scriptErrors 1` and `/etrace`.

**Event Log internals.** `ns:StartEventLog()` sets `ns.diagnostics.logging = true`; Core's dispatcher then calls `ns:LogEvent` for every event it routes. Each entry is a `GetTime()`-stamped string snapshotted immediately — references are never retained, since some events carry frames or tables that would leak or go stale. Bounds: a ring buffer of `EVENT_LOG_SIZE` (500) entries, up to `EVENT_LOG_MAX_ARGS` (8) args each, each capped at `EVENT_LOG_MAX_ARG_LENGTH` (255 bytes — sized to hold a full loot line with its item link). `ns.DIAGNOSTIC_EVENT_EXCLUDE` is the drop list `ns:LogEvent` consults before recording; it is empty because the dispatcher only ever feeds it events the addon registers and none of those is a sustained firehose (the addon listens on the coalesced `BAG_UPDATE_DELAYED`, not raw `BAG_UPDATE`). The lookup stays so a genuine firehose could be excluded later. Pipes are escaped (`|` → `||`) *after* the length cut so links render as literal text in the report editbox instead of as a clickable swatch, and so a mid-link cut can't leave a dangling pipe that eats the next separator.

## Saved Variables

`OpenSesameDB` (account-wide) holds:

- `autoOpen` (boolean, default `true`) — master switch for the auto-open pipeline. Mirrored to `ns.isEnabled` at runtime.
- `speedyLoot` (boolean, default `true`) — gates `ns.HandleSpeedyLoot` (Speedy-Loot.lua), the auto-loot response Core runs on `LOOT_READY`. Mirrored to `ns.isSpeedyLoot`.
- `lootSounds` (boolean, default `false`) — plays a distinct sound on Uncommon+ loot in `CHAT_MSG_LOOT`, but only when the item came from a looted corpse/chest (not from disenchanting, prospecting, or item merges — see the `LOOT_OPENED` source check).
- `lockboxNotifications` (boolean, default `true`) — gates the two lockbox chat notices in Core.lua (`CHAT_MSG_LOOT`) and Speedy-Loot.lua, *in addition to* `autoOpen`. See Lockbox Notifications.
- `showWelcome` (boolean, default `true`) — toggles the login welcome message.
- `showMinimap` (boolean, default `true`) — source of truth for the minimap button's visibility. Mirrored (inverted) onto `minimap.hide` so LibDBIcon obeys it; see Minimap Button.
- `minimap` (table) — owned by LibDBIcon; holds the minimap button position and its `hide` flag. Passed straight to `LDBIcon:Register`. `hide` is a derived mirror of `showMinimap`, not an independent setting.

There is no per-character DB — all settings are account-scoped, and there is no migration chain.

Defaults live in `ns.DEFAULT_CONFIGURATION` (Data/Default-Settings.lua). `EventHandlers:PLAYER_LOGIN` merges them additively: `if ns.DB[key] == nil then ns.DB[key] = value end`. It only fills nil fields and never overrides an explicit user value — `false` is a legitimate stored value, and the `or` form would wipe a deliberately-disabled toggle on every login. Login then derives `minimap.hide` from the merged `showMinimap`, so the LibDBIcon button matches the saved setting.

`ns.ResetSettings` (Core.lua, fired by the **Reset All Open Sesame Options** button under Options → General) is the one place that *overwrites* every `ns.DEFAULT_CONFIGURATION` key back to its default instead of filling nil — so a reset turns the minimap button back on through `showMinimap`. It never wipes the `minimap` subtable; it mirrors the restored `showMinimap` onto `minimap.hide` instead, so the button's saved position survives the reset (wiping it would snap the button back to its default angle). It then re-applies the runtime flags and side effects (`EnsureAutoLoot`, a forced scan, icon refresh) so the reset is live without a `/reload`, and prints `L["MESSAGE_RESET"]` so the player gets confirmation in chat.

## Adding a New Item to the Allowlist

1. Find the item ID on Wowhead Classic (or `/run print(GetItemInfoInstant(...))` in-game).
2. Open `Data/Data.lua` and locate `ns.AllowedItems`.
3. Add a row in the matching expansion block: `[itemId] = true,` for instant-open items, `[itemId] = false,` for lockboxes that need unlocking. End the line with a `-- Item Name` comment for grep-ability.
4. Keep entries alphabetized within the block by item name.
5. If the item is a BoP grab bag that should never auto-loot, add it to `ns.IgnoreItems` with `[itemId] = true,` instead.

No locale changes required — item names aren't localized in the data tables; the inline comments are reference-only.

## Adding a New Locale

Copy `Locales/enUS.lua` to `Locales/<locale>.lua`. Drop the `true` argument from the `NewLocale("Open-Sesame", "<locale>", true)` call (that flag marks the default fallback — only `enUS.lua` sets it). Translate every string. Add the file to the `.toc` immediately after `Locales/enUS.lua`.

Every locale is an independent file in this format. Spanish is no exception: `esES.lua` and `esMX.lua` are separate files even though their strings are usually identical — translate each on its own. Do **not** translate the diagnostics strings; they live in `ns.DiagnosticsStrings`, not in `Locales/`, and are English-only by design.

## Common Pitfalls

- **Master-looter crash in Speedy Loot**: calling `LootSlot()` on an at-or-above-threshold item while the player is the master looter pops `MasterLooterFrame_Show` with no `selectedLootButton` and crashes on a nil `colorInfo`. Speedy-Loot.lua avoids this by standing down entirely when the player is the master looter — the `lootMethod == "master" and masterLooterPartyID == 0` check returns before any `LootSlot()`. Don't remove it.

- **Truthy-fallthrough on availability-picked APIs**: `(C_CVar.GetCVarBool("autoLootDefault")) or (GetCVar(...) == "1")` looks reasonable but falls through to the legacy call whenever the modern API returns `false` (auto-loot off), so the wrong result wins. Pick by availability (`if C_CVar then … else …`), then call exactly one. Same applies to `GetLootMethod` vs `C_PartyInfo.GetLootMethod`.

- **`C_Timer.After` callbacks can't be cancelled**: the `ns.state.openTimerLive` flag in Core.lua prevents double-scheduling the open tick when a scan completes mid-flight. Keep it consistent if you add new entry points into `OpenTick()`.

- **Reading an item ID right after a bag update**: `GetContainerItemID` can return nil immediately after a bag mutation. Use `SafeFastItemID()` (Core.lua), which falls back to parsing the item link with `item:(%d+)`.

- **Pause hysteresis**: pause threshold is `< MIN_FREE_SLOTS`; resume threshold is `< MIN_FREE_SLOTS + 1`. The asymmetry prevents flapping when a single bag slot opens and closes during normal play. Don't collapse them into one comparison.

- **The `OS_ScanTooltip` frame name is intentional**: the scanning tooltip keeps the `OS_` prefix even though the namespace local is `ns`. It's a global frame name — and the source of the `OS_ScanTooltipTextLeft*` font strings `IsItemLocked` reads — so renaming it would break those lookups for no benefit.

- **Event log escaping order**: in `ns:LogEvent`, the 255-byte length cut happens *before* the `|` → `||` escape. Reverse them and a loot line cut mid-link renders as a broken clickable swatch (the classic collapse to a stray `[Sc`) instead of readable text.

- **Resetting must not wipe the `minimap` subtable**: `ns.ResetSettings` overwrites the `ns.DEFAULT_CONFIGURATION` keys back to their defaults but deliberately leaves `OpenSesameDB.minimap` in place, re-deriving only its `hide` field from the restored `showMinimap`. Replacing that with a blanket `OpenSesameDB = {}` (or wiping `minimap`) discards the button's saved position and snaps it back to its default angle. Keep the targeted key overwrite.

- **`minimap.hide` is derived, not authoritative**: write button visibility through `ns.SetMinimapShown`, which keeps `showMinimap` (the source of truth) and `minimap.hide` in sync. Setting `minimap.hide` directly drifts from `showMinimap` and is overwritten on the next `PLAYER_LOGIN`.

- **Lockbox-notification gate must match the UI**: the two lockbox notices are gated on `autoOpen and lockboxNotifications`, and the **Enable Lockbox Notifications** toggle is `hidden` unless `autoOpen` is on. Drop `autoOpen` from a call site (or from the toggle's `hidden` check) and the runtime starts announcing lockboxes the player can no longer see a setting for. Keep both conditions on both Core.lua and Speedy-Loot.lua.

When filing a bug report, include:

- Game version (Classic Era 1.15.x, TBC Anniversary 2.5.x) and locale.
- Class and level.
- Repro steps — what was in your bags, what you were doing, expected vs. actual.
- Any chat output prefixed with `Open Sesame //` and any Lua error text. The Diagnostic Tools panel's Event Log and report buttons produce copy-paste-ready output for this.

PR guidelines:

- Keep changes scoped — one feature or fix per PR.
- Match the existing code style — no `=` alignment padding, no abbreviations, all user-facing strings localized, multi-line comments as `--[[ ]]` blocks.
- Only edit `Locales/enUS.lua`; the other locale files are the maintainer's to translate.
- If you add or rename a SavedVariables field, initialize it in the `PLAYER_LOGIN` merge rather than overwriting existing values.
- If you change item-list behavior (allowlist semantics, ignore-list scope), update this document and `README.md`.
- Update the File Map and Architecture sections here whenever you add a file or move logic between phases.

Every commit and PR description carries a **User Story** — not "I changed X" but who it helps and why:

> *As a [role], I [needed/wanted] [behavior] so that [outcome]. This change [does X].*

For example: *As a player who keeps Speedy Loot on but Auto-Opening off, I wanted looting to work without manually toggling Auto Loot, so that the feature works out of the box. This change calls `ns.EnsureAutoLoot()` whenever either feature is switched on.*
